# Debugging Guide

## Dumping IR and Object Files

Use the following JVM flags to dump LLVM IR and object files generated by Jeandle:

```
java -XX:+JeandleDumpIR -XX:+JeandleDumpObjects \
     -XX:JeandleDumpDirectory=/path/to/output \
     -Xcomp -XX:-TieredCompilation \
     -XX:CompileCommand=compileonly,ClassName::methodName \
     ClassName
```

- `-XX:+JeandleDumpIR` dumps LLVM IR before and after optimization.
- `-XX:+JeandleDumpObjects` dumps the generated object files.
- `-XX:JeandleDumpDirectory=<path>` specifies the output directory (defaults to the working directory of the Java process).

Inspecting the dumped IR and object files helps identify bugs in the compilation pipeline.

## Running and Debugging Jeandle-LLVM Separately

You can run the Jeandle-LLVM optimization pipeline and backend independently from the JVM. This is useful for isolating issues in LLVM passes or code generation.

Jeandle optimization passes query the JVM information through VM callbacks (e.g., type hierarchy queries). To run these passes without the JVM, you need to record VM callback invocations during a JVM run and replay them with `opt`.

### Recording VM Callbacks

Enable recording with `-XX:+JeandleRecordVMCallbacks`. This must be combined with `-XX:+JeandleDumpIR` so the dumped IR can be used with `opt` for replay:

```
java -XX:+JeandleRecordVMCallbacks -XX:+JeandleDumpIR \
     -XX:JeandleDumpDirectory=/path/to/output \
     -Xcomp -XX:-TieredCompilation \
     -XX:CompileCommand=compileonly,ClassName::methodName \
     ClassName
```

For each compiled method, this produces three files in the dump directory:

| File | Content |
| --- | --- |
| `MethodName_timestamp.ll` | LLVM IR before optimization |
| `MethodName_timestamp_optimized.ll` | LLVM IR after optimization |
| `MethodName_timestamp.cblog` | VM callback log (recorded invocations and results) |

The `.cblog` file records each VM callback invocation on a separate line:

```
IsObjectKlass 1 = true
IsSubtype 2 3 = false
IsInterface 3 = true
```

Lines starting with `#` are comments; blank lines are ignored.

### Running Jeandle Optimization Passes

Use the dumped IR and callback log to run the Jeandle optimization pipeline with `opt`. If the pass does not depend on VM callbacks, no callback log is needed:

```
opt --jeandle -S input.ll -o output.ll
```

If the pass depends on VM callbacks, use `--jeandle-vm-callback-log=<file>` with `opt` to replay recorded callbacks. This allows Jeandle passes that depend on VM callbacks to run standalone:

```
opt -S -passes="type-check-elimination" \
    --jeandle-vm-callback-log=/path/to/output/MethodName_timestamp.cblog \
    /path/to/output/MethodName_timestamp.ll
```

You can also run the full Jeandle pipeline:

```
opt --jeandle \
    --jeandle-vm-callback-log=/path/to/output/MethodName_timestamp.cblog \
    -S /path/to/output/MethodName_timestamp.ll
```

During replay, callback entries are consumed sequentially. If the log is exhausted or a mismatch occurs, a warning is printed to stderr and a safe default value is returned (`false` for bool, `0` for uintptr_t/int).

### Running the LLVM Backend

Use `llc` to run code generation on the dumped IR:

```
llc input.ll -o output.s
```

### Debugging Jeandle-LLVM Passes

Since `opt` and `llc` are standalone executables, you can set breakpoints directly in Jeandle-LLVM pass code using a debugger:

```
gdb --args opt --jeandle -S input.ll -o output.ll
```
```
(gdb) break llvm::JeandlePassName::run
(gdb) run
```

This makes it easy to step through individual optimization passes without running the entire JVM.

## Inspecting Linked Code with PrintAssembly

Use `-XX:+PrintAssembly` to inspect the linked code's structure, including assembly, relocations, and code stubs:

```
java -XX:+PrintAssembly \
     -Xcomp -XX:-TieredCompilation \
     -XX:CompileCommand=compileonly,ClassName::methodName \
     ClassName
```

This shows how the compiled code is laid out in HotSpot's Code Cache after linking and relocation.

## Adding Breakpoints to Jeandle Compiled Code

To set breakpoints directly in Jeandle compiled code:

1. Run with `-XX:+PrintAssembly` and set a debugger breakpoint at `JeandleCompilation::JeandleCompilation`:
   ```
   gdb --args java -XX:+PrintAssembly \
        -Xcomp -XX:-TieredCompilation \
        -XX:CompileCommand=compileonly,ClassName::methodName \
        ClassName
   ```
   ```
   (gdb) break JeandleCompilation::JeandleCompilation
   (gdb) run
   ```

2. When the breakpoint hits and `JeandleCompilation::JeandleCompilation` finishes, the PrintAssembly output shows the compiled assembly along with the code address.

3. Use the code address to set a breakpoint directly on the Jeandle compiled code:
   ```
   (gdb) break *0x<code_address>
   (gdb) continue
   ```

This allows stepping through the actual machine code generated by Jeandle.
